Self-Attention Mechanism

How tokens learn to look at each other — from fuzzy dictionary lookups to multi-head attention and efficient inference with KV caches.

<\!-- ==================== INTUITION LAYER ==================== -->

1.1 Intuition Layer

Intuition

1.1.1 Attention as a Soft Dictionary Lookup

Imagine you walk into a library looking for information about "climate change effects on agriculture." You have a query — what you're looking for. Each book on the shelf has a key — its title and topic. And each book contains a value — its actual content.

In a traditional (hard) dictionary lookup, you find the exact matching key and retrieve its value. But what if no single book perfectly matches your query? In a soft lookup, you compare your query against all keys, compute a relevance score for each, and then take a weighted blend of all values — books more relevant to your query contribute more to the answer.

That is exactly what attention does. Every token in a sequence computes a query ("what am I looking for?"), and every token also provides a key ("what do I contain?") and a value ("here's my information"). The attention mechanism computes similarity between queries and keys, then uses those similarities to create a weighted sum of values.

Key Idea
Attention is a soft dictionary lookup: compute relevance of every key to your query, then blend all values proportional to relevance. No single item is selected — everything contributes, weighted by similarity.
<\!-- SVG: Soft Dictionary Lookup -->
Figure 1.1: Attention as a soft dictionary lookup. The query is compared to all keys (similarity scores), normalized via softmax (weights), then used to create a weighted sum of values.

1.1.2 Why Tokens Need to Talk to Each Other

Consider the sentence: "The animal didn't cross the street because it was too tired."

What does "it" refer to? As a human, you instantly know it refers to "the animal" — because an animal can be tired, but a street cannot. But to arrive at this conclusion, the representation of "it" needs to incorporate information from "animal." The token "it" needs to "look at" other tokens in the sentence and determine which ones are relevant.

This is precisely what self-attention enables: every token in the sequence gets to look at every other token and decide how much information to pull from each. The result is a new representation for each token that is context-aware — it incorporates relevant information from the entire sequence.

1.1.3 Attention as Information Routing

Another way to think about attention: it's a learned routing mechanism. Each token broadcasts its information (value), and each token also decides which broadcasts to listen to (via query-key matching). The attention weights form a routing table — a matrix that specifies how much information flows from every source token to every destination token.

<\!-- ==================== FORMALISM LAYER ==================== -->

1.2 Formalism Layer

Formalism

1.2.1 The QKV Framework

Now let's formalize. Given an input sequence $X \in \mathbb{R}^{n \times d_{\text{model}}}$ (where $n$ is the sequence length and $d_{\text{model}}$ is the embedding dimension), we compute three projections:

$$Q = XW^Q, \quad K = XW^K, \quad V = XW^V$$

where $W^Q, W^K \in \mathbb{R}^{d_{\text{model}} \times d_k}$ and $W^V \in \mathbb{R}^{d_{\text{model}} \times d_v}$. These learned projection matrices transform each token into three roles:

  • Query ($Q$): "What am I looking for?" — each row is a query vector
  • Key ($K$): "What do I contain?" — each row is a key vector
  • Value ($V$): "Here's my information" — each row is a value vector

The core attention operation is then:

$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right) V$$

Let's trace the dimensions step by step:

StepOperationShape
1$QK^T$$(n \times d_k) \cdot (d_k \times n) = n \times n$
2Scale by $\frac{1}{\sqrt{d_k}}$$n \times n$
3Softmax (row-wise)$n \times n$ (each row sums to 1)
4Multiply by $V$$(n \times n) \cdot (n \times d_v) = n \times d_v$

The $n \times n$ matrix from step 3 is the attention weight matrix — it tells us how much each token attends to every other token. Each row corresponds to a query token, and each column to a key token.

1.2.2 The Scaling Factor: Why $\frac{1}{\sqrt{d_k}}$?

This seemingly minor detail is crucial. Here's why we need it:

Assume the components of $q$ and $k$ are independent random variables with zero mean and unit variance. The dot product $q \cdot k = \sum_{i=1}^{d_k} q_i k_i$ is a sum of $d_k$ independent terms, each with zero mean and variance 1. By the properties of variance:

$$\text{Var}(q \cdot k) = \sum_{i=1}^{d_k} \text{Var}(q_i k_i) = \sum_{i=1}^{d_k} 1 = d_k$$

So the variance of the dot products grows linearly with $d_k$. For $d_k = 64$, the standard deviation of dot products is $\sqrt{64} = 8$. This pushes some attention logits to very large values, which causes softmax to produce near-one-hot distributions — essentially ignoring all but one token.

Why This Matters
Without the $\frac{1}{\sqrt{d_k}}$ scaling, attention weights become nearly one-hot for large dimensions. This kills gradient flow (softmax gradients vanish at extremes) and prevents the model from learning nuanced attention patterns. Dividing by $\sqrt{d_k}$ normalizes variance back to 1.
<\!-- SVG: Softmax with and without scaling -->
Figure 1.3: Softmax output without scaling (left) produces a near-one-hot "winner takes all" distribution. With $1/\sqrt{d_k}$ scaling (right), the distribution is smooth and informative.

1.2.3 Softmax and the Attention Distribution

The softmax function converts raw scores (logits) into a probability distribution:

$$\text{softmax}(z_i) = \frac{e^{z_i}}{\sum_{j=1}^{n} e^{z_j}}$$

Key properties: all outputs are positive, they sum to 1, and larger inputs get exponentially more weight. The temperature variant $\text{softmax}(z/T)$ controls sharpness — low $T$ makes it peakier (approaching argmax), high $T$ makes it more uniform.

1.2.4 Multi-Head Attention

Instead of performing a single attention function with $d_{\text{model}}$-dimensional keys, queries, and values, the Transformer uses multiple attention heads in parallel, each operating on a lower-dimensional projection:

$$\text{MultiHead}(Q, K, V) = \text{Concat}(\text{head}_1, \ldots, \text{head}_h) W^O$$ $$\text{where } \text{head}_i = \text{Attention}(QW_i^Q, KW_i^K, VW_i^V)$$

With $h$ heads, each head operates on $d_k = d_v = d_{\text{model}} / h$ dimensions. For the standard config ($d_{\text{model}} = 512$, $h = 8$): each head uses $d_k = 64$ dimensions.

Why multiple heads? Different heads can learn different types of relationships:

  • One head might learn positional patterns (attend to adjacent tokens)
  • Another might learn syntactic patterns (subject-verb agreement)
  • Another might learn semantic patterns (coreference resolution)

The output projection $W^O \in \mathbb{R}^{hd_v \times d_{\text{model}}}$ recombines the heads. Total parameter count for MHA: $4 \times d_{\text{model}}^2$ (for $W^Q, W^K, W^V, W^O$).

<\!-- SVG: Multi-Head Attention -->
Figure 1.4: Multi-head attention. Input is projected into h parallel heads, each computing attention on d_k dimensions. Outputs are concatenated and projected back to d_model.

1.2.5 Self-Attention vs Cross-Attention vs Causal Attention

The attention mechanism has three main variants, distinguished by where Q, K, V come from and what mask is applied:

TypeQ SourceK, V SourceMaskUsed In
Self-AttentionSame sequenceSame sequenceNone (bidirectional)Encoder (BERT)
Causal Self-AttentionSame sequenceSame sequenceLower-triangularDecoder (GPT)
Cross-AttentionDecoderEncoder outputNoneEncoder-Decoder →

The causal mask is a lower-triangular matrix that prevents token $i$ from attending to any token $j > i$ (future positions). This is essential for autoregressive generation — the model can only use past context when predicting the next token.

Figure 1.5: Causal attention mask. Green cells (1) are allowed; red cells (0) are masked with $-\infty$ before softmax. Token $t_3$ can attend to $t_1, t_2, t_3$ but not $t_4$.

1.2.6 Complexity Analysis

The computational bottleneck of self-attention:

ResourceComplexityNotes
Time (FLOPs)$O(n^2 \cdot d)$$QK^T$ is $O(n^2 d)$, matmul with $V$ is $O(n^2 d)$
Memory$O(n^2)$Storing the full $n \times n$ attention matrix
For $n=2048, d=512$~4M entriesManageable
For $n=32768, d=512$~1B entriesProblematic\!

This quadratic scaling is the fundamental bottleneck of Transformers for long sequences. Solutions include sparse attention, linear attention, and FlashAttention — all covered in Modern Variants →.

1.3 Engineering Layer

Engineering

1.3.1 Scaled Dot-Product Attention

Python — scaled_dot_product_attention
import torch
import torch.nn.functional as F

def scaled_dot_product_attention(Q, K, V, mask=None):
    """
    Compute scaled dot-product attention.
    
    Args:
        Q: Queries (batch, heads, seq_len, d_k)
        K: Keys    (batch, heads, seq_len, d_k)
        V: Values  (batch, heads, seq_len, d_v)
        mask: Optional mask (broadcastable to attention shape)
    
    Returns:
        output: Weighted sum of values (batch, heads, seq_len, d_v)
        weights: Attention weights (batch, heads, seq_len, seq_len)
    """
    d_k = Q.size(-1)
    
    # Step 1: Compute raw attention scores
    scores = torch.matmul(Q, K.transpose(-2, -1)) / (d_k ** 0.5)
    
    # Step 2: Apply mask (if provided)
    if mask is not None:
        scores = scores.masked_fill(mask == 0, float('-inf'))
    
    # Step 3: Softmax to get attention weights
    weights = F.softmax(scores, dim=-1)
    
    # Step 4: Weighted sum of values
    output = torch.matmul(weights, V)
    
    return output, weights

1.3.2 Multi-Head Attention

Python — MultiHeadAttention module
import torch
import torch.nn as nn

class MultiHeadAttention(nn.Module):
    def __init__(self, d_model, n_heads, dropout=0.1):
        super().__init__()
        assert d_model % n_heads == 0, "d_model must be divisible by n_heads"
        
        self.d_model = d_model
        self.n_heads = n_heads
        self.d_k = d_model // n_heads
        
        # Q, K, V projections (combined into single matrices for efficiency)
        self.W_q = nn.Linear(d_model, d_model, bias=False)
        self.W_k = nn.Linear(d_model, d_model, bias=False)
        self.W_v = nn.Linear(d_model, d_model, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)
        
        self.dropout = nn.Dropout(dropout)
    
    def split_heads(self, x, batch_size):
        """Reshape (batch, seq, d_model) -> (batch, heads, seq, d_k)"""
        x = x.view(batch_size, -1, self.n_heads, self.d_k)
        return x.transpose(1, 2)
    
    def forward(self, x, mask=None):
        batch_size = x.size(0)
        
        # Project to Q, K, V
        Q = self.split_heads(self.W_q(x), batch_size)
        K = self.split_heads(self.W_k(x), batch_size)
        V = self.split_heads(self.W_v(x), batch_size)
        
        # Compute attention
        d_k = Q.size(-1)
        scores = torch.matmul(Q, K.transpose(-2, -1)) / (d_k ** 0.5)
        
        if mask is not None:
            scores = scores.masked_fill(mask == 0, float('-inf'))
        
        weights = F.softmax(scores, dim=-1)
        weights = self.dropout(weights)
        
        attn_output = torch.matmul(weights, V)
        
        # Concatenate heads: (batch, heads, seq, d_k) -> (batch, seq, d_model)
        attn_output = attn_output.transpose(1, 2).contiguous()
        attn_output = attn_output.view(batch_size, -1, self.d_model)
        
        # Final linear projection
        return self.W_o(attn_output)


# Usage example
mha = MultiHeadAttention(d_model=512, n_heads=8)
x = torch.randn(2, 10, 512)  # batch=2, seq_len=10, d_model=512
output = mha(x)
print(f"Input:  {x.shape}")   # [2, 10, 512]
print(f"Output: {output.shape}")  # [2, 10, 512]

1.3.3 Causal Mask Implementation

Python — Causal mask
def create_causal_mask(seq_len, device='cpu'):
    """Create a lower-triangular causal mask.
    
    Returns a mask of shape (1, 1, seq_len, seq_len) for broadcasting
    across batch and head dimensions.
    """
    mask = torch.tril(torch.ones(seq_len, seq_len, device=device))
    return mask.unsqueeze(0).unsqueeze(0)  # (1, 1, seq, seq)

# Example: mask for sequence length 5
mask = create_causal_mask(5)
print(mask.squeeze())

1.3.4 KV Cache for Efficient Inference

During autoregressive generation, we predict one token at a time. Without optimization, at step $t$ we'd recompute attention over all $t$ tokens — wasteful because the K and V for previous tokens don't change.

The KV cache stores previously computed K and V tensors. At each new step, we only compute K/V for the new token and append them to the cache:

Python — KV cache
class CachedMultiHeadAttention(nn.Module):
    def __init__(self, d_model, n_heads):
        super().__init__()
        self.n_heads = n_heads
        self.d_k = d_model // n_heads
        self.W_q = nn.Linear(d_model, d_model, bias=False)
        self.W_k = nn.Linear(d_model, d_model, bias=False)
        self.W_v = nn.Linear(d_model, d_model, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)
    
    def forward(self, x, cache=None):
        B, T, C = x.shape
        
        Q = self.W_q(x).view(B, T, self.n_heads, self.d_k).transpose(1, 2)
        K = self.W_k(x).view(B, T, self.n_heads, self.d_k).transpose(1, 2)
        V = self.W_v(x).view(B, T, self.n_heads, self.d_k).transpose(1, 2)
        
        if cache is not None:
            # Append new K, V to cached K, V
            K = torch.cat([cache['k'], K], dim=2)
            V = torch.cat([cache['v'], V], dim=2)
        
        # Update cache for next step
        new_cache = {'k': K, 'v': V}
        
        # Attention: Q is only for new tokens, K/V include full history
        scores = torch.matmul(Q, K.transpose(-2, -1)) / (self.d_k ** 0.5)
        weights = F.softmax(scores, dim=-1)
        output = torch.matmul(weights, V)
        
        output = output.transpose(1, 2).contiguous().view(B, T, C)
        return self.W_o(output), new_cache
KV Cache Memory Cost
Memory per token: $2 \times n_{\text{layers}} \times d_{\text{model}} \times \text{bytes}$. For a 7B model (32 layers, $d=4096$, FP16): $2 \times 32 \times 4096 \times 2 = 512\text{KB}$ per token. For a 2048-token context: ~1GB of KV cache alone. This is why GQA matters\!

1.3.5 Multi-Query and Grouped-Query Attention

The KV cache is the dominant memory cost during inference. Two techniques reduce it:

MethodQ HeadsKV HeadsKV Cache SizeQuality
MHA (Multi-Head)$h$$h$1× (baseline)Best
MQA (Multi-Query)$h$1$1/h$×Slight degradation
GQA (Grouped-Query)$h$$g$ (groups)$g/h$×Near-MHA quality

GQA (used in LLaMA 2 70B with $h=64, g=8$): groups of 8 query heads share one KV head, reducing KV cache by 8×.

Python — GQA head grouping
class GroupedQueryAttention(nn.Module):
    def __init__(self, d_model, n_q_heads, n_kv_heads):
        super().__init__()
        self.n_q_heads = n_q_heads
        self.n_kv_heads = n_kv_heads
        self.n_groups = n_q_heads // n_kv_heads  # queries per KV head
        self.d_k = d_model // n_q_heads
        
        self.W_q = nn.Linear(d_model, n_q_heads * self.d_k, bias=False)
        self.W_k = nn.Linear(d_model, n_kv_heads * self.d_k, bias=False)
        self.W_v = nn.Linear(d_model, n_kv_heads * self.d_k, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)
    
    def forward(self, x):
        B, T, _ = x.shape
        
        Q = self.W_q(x).view(B, T, self.n_q_heads, self.d_k).transpose(1, 2)
        K = self.W_k(x).view(B, T, self.n_kv_heads, self.d_k).transpose(1, 2)
        V = self.W_v(x).view(B, T, self.n_kv_heads, self.d_k).transpose(1, 2)
        
        # Repeat KV heads to match Q heads: (B, n_kv, T, d) -> (B, n_q, T, d)
        K = K.repeat_interleave(self.n_groups, dim=1)
        V = V.repeat_interleave(self.n_groups, dim=1)
        
        scores = torch.matmul(Q, K.transpose(-2, -1)) / (self.d_k ** 0.5)
        weights = F.softmax(scores, dim=-1)
        output = torch.matmul(weights, V)
        
        output = output.transpose(1, 2).contiguous().view(B, T, -1)
        return self.W_o(output)


# LLaMA 2 70B config: 64 query heads, 8 KV heads
gqa = GroupedQueryAttention(d_model=8192, n_q_heads=64, n_kv_heads=8)
print(f"KV cache reduction: {64/8}x = 8x smaller than MHA")

1.4 Going Deeper

What Different Heads Learn

Research has shown that attention heads specialize. Some heads learn syntactic dependencies (subject-verb), some learn positional patterns (attend to previous token), and some learn rare word relationships. Tools like BertViz can visualize these patterns.

Attention Entropy and Collapse

The entropy of attention weights measures how "spread out" or "focused" attention is. Very low entropy (near-one-hot) means the head is only looking at one token. Very high entropy (near-uniform) means the head provides no useful routing. Monitoring attention entropy during training can diagnose attention collapse.
