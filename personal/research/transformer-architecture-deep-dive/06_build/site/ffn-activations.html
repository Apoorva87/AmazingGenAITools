Feed-Forward Networks & Activation Functions

The silent workhorse of the Transformer: how position-wise feed-forward layers transform token representations, store factual knowledge, and why the choice of activation function matters more than you might expect.

Intuition Formalism Engineering

4.1 Intuition: What Happens Between Attention Layers?

The Role of the FFN

Self-attention is the headline act of the Transformer, but it only does one thing: it mixes information across tokens. After attention has decided which tokens should talk to which, the feed-forward network (FFN) processes each token's representation independently, applying the same learned non-linear transformation to every position.

Think of it this way: attention is a communication step (tokens share information) and the FFN is a computation step (each token digests what it learned). Together they form the two fundamental operations inside every Transformer block, repeated layer after layer.

Key Idea
Attention = inter-token mixing (who should I listen to?). FFN = intra-token transformation (what should I become?). Every Transformer layer applies both in sequence.

The Expansion-Compression Metaphor

The FFN has a distinctive bottleneck shape. The input has dimension \(d_{\text{model}}\). The FFN first expands it to a hidden dimension \(d_{\text{ff}} = 4 \times d_{\text{model}}\), applies a non-linearity, then compresses it back to \(d_{\text{model}}\).

Why expand? In a higher-dimensional space, non-linear transformations can carve out more complex decision boundaries. The expansion gives the network room to work, and the compression forces it to distill the result back into a compact representation.

Parameter Budget
The FFN accounts for roughly two-thirds of all parameters in a standard Transformer layer. For a layer with \(d_{\text{model}} = 512\) and \(d_{\text{ff}} = 2048\), the FFN has \(2 \times 512 \times 2048 \approx 2\text{M}\) parameters, while multi-head attention has about \(4 \times 512^2 \approx 1\text{M}\).
Figure 4.1 — The FFN bottleneck: expand to \(4 \times d_{\text{model}}\), apply non-linearity, compress back.

This expand-compress pattern is not unique to Transformers — autoencoders and bottleneck residual blocks in ResNets use similar ideas. What makes the Transformer FFN special is that it is applied identically and independently to every token position, making it trivially parallelizable.

4.2 Formalism

4.2.1 Position-wise Feed-Forward Network

Given an input vector \(\mathbf{x} \in \mathbb{R}^{d_{\text{model}}}\) at a single token position, the position-wise FFN computes:

\[\text{FFN}(\mathbf{x}) = \mathbf{W}_2 \, \sigma\!\left(\mathbf{W}_1 \mathbf{x} + \mathbf{b}_1\right) + \mathbf{b}_2\] (4.1)

where:

  • \(\mathbf{W}_1 \in \mathbb{R}^{d_{\text{ff}} \times d_{\text{model}}}\) — expansion weight matrix
  • \(\mathbf{W}_2 \in \mathbb{R}^{d_{\text{model}} \times d_{\text{ff}}}\) — compression weight matrix
  • \(\mathbf{b}_1 \in \mathbb{R}^{d_{\text{ff}}}\), \(\mathbf{b}_2 \in \mathbb{R}^{d_{\text{model}}}\) — bias vectors
  • \(\sigma\) — a non-linear activation function (ReLU in the original paper)

The standard setting is \(d_{\text{ff}} = 4 \times d_{\text{model}}\). For the base Transformer (\(d_{\text{model}} = 512\)), this gives \(d_{\text{ff}} = 2048\). The word position-wise means the same weight matrices are applied independently at each position — no information crosses token boundaries inside the FFN.

Parameter Count
Each FFN layer has \(2 \times d_{\text{model}} \times d_{\text{ff}}\) weight parameters (plus biases). For the base model: \(2 \times 512 \times 2048 = 2{,}097{,}152 \approx 2\text{M}\) per layer. With 6 encoder + 6 decoder layers, that is about 25M parameters just from FFNs.

4.2.2 Activation Functions

The choice of activation function \(\sigma\) has a surprisingly large effect on model quality. Here are the three most important ones for Transformers:

ReLU (Rectified Linear Unit)

\[\text{ReLU}(x) = \max(0, x)\] (4.2)

Used in the original Transformer (Vaswani et al., 2017). Computationally cheap, introduces sparsity (many activations are exactly zero), but suffers from the dying ReLU problem: neurons whose inputs are always negative will never activate and their gradients are perpetually zero.

GELU (Gaussian Error Linear Unit)

\[\text{GELU}(x) = x \cdot \Phi(x)\] (4.3)

where \(\Phi(x)\) is the CDF of the standard normal distribution. The intuition: multiply each input by the probability that it is positive under a Gaussian. This gives a smooth, non-monotonic "soft gate" around zero. GELU is used in BERT, GPT-2, and many subsequent models. It can be approximated as \(\text{GELU}(x) \approx 0.5 x \left(1 + \tanh\!\left[\sqrt{2/\pi}\,(x + 0.044715\,x^3)\right]\right)\).

SiLU / Swish

\[\text{SiLU}(x) = x \cdot \sigma(x) = \frac{x}{1 + e^{-x}}\] (4.4)

Also called Swish (Ramachandran et al., 2017). It is self-gated: the input is multiplied by its own sigmoid. Like GELU, it is smooth and allows small negative values to pass through. Used in PaLM and LLaMA.

Figure 4.2 — ReLU, GELU, and SiLU/Swish: function values (top) and derivatives (bottom). Note how GELU and SiLU allow small negative values through and have smooth derivatives everywhere.
Activation Formula Smooth? Sparse? Used In
ReLU \(\max(0, x)\) No (kink at 0) Yes Original Transformer
GELU \(x \cdot \Phi(x)\) Yes Approximately BERT, GPT-2, GPT-3
SiLU/Swish \(x \cdot \sigma(x)\) Yes No PaLM, LLaMA

4.2.3 Gated Linear Units (GLU) and SwiGLU

A powerful variant replaces the simple FFN with a gated architecture. Instead of one up-projection, the network uses two parallel projections: one produces a "value" and the other produces a "gate" that controls how much of the value to let through.

GLU (Dauphin et al., 2017)

\[\text{GLU}(\mathbf{x}) = \left(\mathbf{x}\mathbf{W}_1\right) \odot \sigma\!\left(\mathbf{x}\mathbf{W}_3\right)\] (4.5)

where \(\odot\) denotes element-wise multiplication and \(\sigma\) is the sigmoid function. The idea is that \(\mathbf{x}\mathbf{W}_3\) learns which dimensions to activate, while \(\mathbf{x}\mathbf{W}_1\) provides the content to activate.

SwiGLU (Shazeer, 2020)

\[\text{SwiGLU}(\mathbf{x}) = \left(\text{SiLU}(\mathbf{x}\mathbf{W}_1)\right) \odot \left(\mathbf{x}\mathbf{W}_3\right)\] (4.6)

The full SwiGLU FFN then applies a down-projection:

\[\text{FFN}_{\text{SwiGLU}}(\mathbf{x}) = \left(\text{SiLU}(\mathbf{x}\mathbf{W}_1) \odot \mathbf{x}\mathbf{W}_3\right) \mathbf{W}_2\] (4.7)

SwiGLU uses three weight matrices: \(\mathbf{W}_1\) (gate projection), \(\mathbf{W}_3\) (value/up projection), and \(\mathbf{W}_2\) (down projection). Since we now have three matrices instead of two, keeping the total parameter count roughly constant requires reducing \(d_{\text{ff}}\). The standard choice is:

\[d_{\text{ff}} = \frac{2}{3} \times 4 \times d_{\text{model}} \approx 2.67 \times d_{\text{model}}\] (4.8)
Shazeer (2020)
In "GLU Variants Improve Transformer," Noam Shazeer systematically compared activation functions in Transformers. SwiGLU consistently outperformed standard ReLU and GELU across multiple benchmarks, leading to its adoption in PaLM, LLaMA, Mistral, and most modern LLMs.
Figure 4.3 — SwiGLU dual-path architecture: the gate path controls element-wise which hidden dimensions pass through.

4.2.4 FFN as Key-Value Memory

A fascinating perspective from Geva et al. (2021) reinterprets the FFN as a form of key-value memory. Consider the FFN written as a sum over hidden units:

\[\text{FFN}(\mathbf{x}) = \sum_{i=1}^{d_{\text{ff}}} \underbrace{\sigma(\mathbf{w}_{1,i} \cdot \mathbf{x})}_{\text{activation (match score)}} \cdot \underbrace{\mathbf{w}_{2,i}}_{\text{value vector}}\] (4.9)

where:

  • \(\mathbf{w}_{1,i}\) is the \(i\)-th row of \(\mathbf{W}_1\) — it acts as a key that "activates" when the input matches a particular pattern
  • \(\mathbf{w}_{2,i}\) is the \(i\)-th column of \(\mathbf{W}_2\) — it acts as the value to inject into the residual stream
  • The activation \(\sigma(\mathbf{w}_{1,i} \cdot \mathbf{x})\) is the "match score" between the input and the \(i\)-th key
FFN as Memory
The FFN is remarkably similar to attention! In attention, queries match against keys to produce weights over values. In the FFN, the input matches against learned key vectors (rows of \(\mathbf{W}_1\)) to produce weights over value vectors (columns of \(\mathbf{W}_2\)). The crucial difference: attention retrieves from the context (other tokens), while FFN retrieves from learned parameters (static memory).

This interpretation has a profound implication: FFN layers store factual knowledge in their weights, while attention layers route information between tokens. This is supported by empirical work showing that knowledge-related facts (e.g., "The Eiffel Tower is in Paris") are localized in specific FFN neurons, and can be edited by modifying specific rows/columns of the FFN matrices.

Figure 4.4 — FFN as key-value memory (Geva et al., 2021). Each hidden neuron acts as a key-value pair: keys match input patterns, values inject corresponding information.

4.3 Engineering

4.3.1 Standard FFN Module

The standard position-wise feed-forward network as described in "Attention Is All You Need." We implement it with configurable activation and optional dropout for regularization.

Python — standard_ffn.py 
import torch
import torch.nn as nn
import torch.nn.functional as F
from typing import Optional


class PositionWiseFFN(nn.Module):
    """Standard position-wise feed-forward network.

    FFN(x) = W2 * activation(W1 * x + b1) + b2

    Args:
        d_model: Input/output dimension (e.g. 512).
        d_ff: Hidden (expanded) dimension (e.g. 2048).
        activation: One of 'relu', 'gelu', 'silu'.
        dropout: Dropout probability applied after the activation.
        bias: Whether to include bias terms in linear layers.
    """

    def __init__(
        self,
        d_model: int = 512,
        d_ff: int = 2048,
        activation: str = "relu",
        dropout: float = 0.1,
        bias: bool = True,
    ):
        super().__init__()
        self.w1 = nn.Linear(d_model, d_ff, bias=bias)     # Expansion
        self.w2 = nn.Linear(d_ff, d_model, bias=bias)      # Compression
        self.dropout = nn.Dropout(dropout)

        # Select activation function
        activations = {
            "relu": F.relu,
            "gelu": F.gelu,
            "silu": F.silu,
        }
        if activation not in activations:
            raise ValueError(
                f"Unknown activation '{activation}'. "
                f"Choose from {list(activations.keys())}"
            )
        self.activation = activations[activation]

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        """Apply position-wise FFN to each token independently.

        Args:
            x: Input tensor of shape (batch, seq_len, d_model).

        Returns:
            Output tensor of shape (batch, seq_len, d_model).
        """
        # x:        (batch, seq_len, d_model)
        # hidden:   (batch, seq_len, d_ff)       -- expansion
        # output:   (batch, seq_len, d_model)    -- compression
        hidden = self.activation(self.w1(x))
        hidden = self.dropout(hidden)
        output = self.w2(hidden)
        return output


# ---- Quick test ----
if __name__ == "__main__":
    batch, seq_len, d_model, d_ff = 2, 10, 512, 2048
    ffn = PositionWiseFFN(d_model, d_ff, activation="gelu")
    x = torch.randn(batch, seq_len, d_model)
    out = ffn(x)
    print(f"Input:  {x.shape}")   # (2, 10, 512)
    print(f"Output: {out.shape}") # (2, 10, 512)
    total_params = sum(p.numel() for p in ffn.parameters())
    print(f"Parameters: {total_params:,}")  # ~2.1M

4.3.2 SwiGLU FFN Module

The SwiGLU variant used in LLaMA, Mistral, and most modern LLMs. Note the three weight matrices and the reduced \(d_{\text{ff}}\) to match the parameter budget of the standard FFN.

Python — swiglu_ffn.py 
import torch
import torch.nn as nn
import torch.nn.functional as F
import math


class SwiGLUFFN(nn.Module):
    """SwiGLU feed-forward network (Shazeer, 2020).

    FFN_SwiGLU(x) = (SiLU(x @ W_gate) * (x @ W_up)) @ W_down

    Uses three weight matrices:
      - W_gate (d_model -> d_ff): produces gate activations
      - W_up   (d_model -> d_ff): produces value activations
      - W_down (d_ff -> d_model): projects back down

    The hidden dimension d_ff is typically set to (2/3) * 4 * d_model
    (rounded to a multiple of 256 for hardware efficiency) to keep
    the total parameter count comparable to a standard FFN.

    Args:
        d_model: Input/output dimension.
        d_ff: Hidden dimension. If None, uses (2/3)*4*d_model rounded
              to the nearest multiple of `multiple_of`.
        dropout: Dropout probability.
        bias: Whether to use bias in linear layers (usually False
              in modern LLMs).
        multiple_of: Round d_ff to this multiple for GPU efficiency.
    """

    def __init__(
        self,
        d_model: int = 512,
        d_ff: int | None = None,
        dropout: float = 0.0,
        bias: bool = False,
        multiple_of: int = 256,
    ):
        super().__init__()

        # Compute hidden dimension if not provided
        if d_ff is None:
            d_ff = int(2 * (4 * d_model) / 3)
            # Round up to nearest multiple for hardware efficiency
            d_ff = multiple_of * ((d_ff + multiple_of - 1) // multiple_of)

        self.d_model = d_model
        self.d_ff = d_ff

        # Three projection matrices
        self.w_gate = nn.Linear(d_model, d_ff, bias=bias)  # Gate
        self.w_up   = nn.Linear(d_model, d_ff, bias=bias)  # Value/Up
        self.w_down = nn.Linear(d_ff, d_model, bias=bias)  # Down
        self.dropout = nn.Dropout(dropout)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        """Apply SwiGLU FFN.

        Args:
            x: (batch, seq_len, d_model)

        Returns:
            (batch, seq_len, d_model)
        """
        # Gate path:  SiLU(x @ W_gate)       -> (batch, seq, d_ff)
        # Value path: x @ W_up               -> (batch, seq, d_ff)
        # Combined:   gate * value            -> (batch, seq, d_ff)
        # Output:     combined @ W_down       -> (batch, seq, d_model)
        gate = F.silu(self.w_gate(x))
        value = self.w_up(x)
        hidden = gate * value  # element-wise product
        hidden = self.dropout(hidden)
        output = self.w_down(hidden)
        return output


# ---- Quick test ----
if __name__ == "__main__":
    d_model = 512
    ffn = SwiGLUFFN(d_model=d_model)
    print(f"d_model={d_model}, d_ff={ffn.d_ff}")

    x = torch.randn(2, 10, d_model)
    out = ffn(x)
    print(f"Input:  {x.shape}")
    print(f"Output: {out.shape}")

    total = sum(p.numel() for p in ffn.parameters())
    print(f"Parameters: {total:,}")
    print(f"  w_gate: {ffn.w_gate.weight.numel():,}")
    print(f"  w_up:   {ffn.w_up.weight.numel():,}")
    print(f"  w_down: {ffn.w_down.weight.numel():,}")

4.3.3 Activation Comparison Code

A self-contained script that plots all three activation functions and their derivatives side by side, matching the SVG diagram above but with pixel-accurate numerical evaluation.

Python — plot_activations.py 
import torch
import matplotlib.pyplot as plt
import numpy as np

def plot_activations():
    """Plot ReLU, GELU, and SiLU with their derivatives."""
    x = torch.linspace(-5, 5, 1000, requires_grad=False)

    # --- Compute activation values ---
    relu_y = torch.relu(x)
    gelu_y = torch.nn.functional.gelu(x)
    silu_y = torch.nn.functional.silu(x)

    # --- Compute derivatives via autograd ---
    def derivative(fn, x_vals):
        x_vals = x_vals.clone().requires_grad_(True)
        y = fn(x_vals)
        y.sum().backward()
        return x_vals.grad.detach()

    relu_dy = derivative(torch.relu, x)
    gelu_dy = derivative(torch.nn.functional.gelu, x)
    silu_dy = derivative(torch.nn.functional.silu, x)

    x_np = x.numpy()

    fig, axes = plt.subplots(2, 1, figsize=(10, 8), sharex=True)
    fig.suptitle("Activation Functions in Transformers", fontsize=14, y=0.95)

    colors = {"ReLU": "#E74C3C", "GELU": "#3498DB", "SiLU": "#2ECC71"}

    # Top: function values
    ax = axes[0]
    ax.plot(x_np, relu_y.numpy(), color=colors["ReLU"], lw=2.5, label="ReLU")
    ax.plot(x_np, gelu_y.numpy(), color=colors["GELU"], lw=2.5, label="GELU")
    ax.plot(x_np, silu_y.numpy(), color=colors["SiLU"], lw=2.5, label="SiLU/Swish")
    ax.axhline(0, color="gray", lw=0.5)
    ax.axvline(0, color="gray", lw=0.5)
    ax.set_ylabel("f(x)")
    ax.set_ylim(-1.5, 5.5)
    ax.legend(loc="upper left")
    ax.set_title("Function Values")
    ax.grid(True, alpha=0.3)

    # Bottom: derivatives
    ax = axes[1]
    ax.plot(x_np, relu_dy.numpy(), color=colors["ReLU"], lw=2.5, label="ReLU'")
    ax.plot(x_np, gelu_dy.numpy(), color=colors["GELU"], lw=2.5, label="GELU'")
    ax.plot(x_np, silu_dy.numpy(), color=colors["SiLU"], lw=2.5, label="SiLU'")
    ax.axhline(0, color="gray", lw=0.5)
    ax.axhline(1, color="gray", lw=0.5, ls="--")
    ax.axvline(0, color="gray", lw=0.5)
    ax.set_ylabel("f'(x)")
    ax.set_xlabel("x")
    ax.set_ylim(-0.3, 1.5)
    ax.legend(loc="upper left")
    ax.set_title("Derivatives")
    ax.grid(True, alpha=0.3)

    plt.tight_layout()
    plt.savefig("activation_comparison.png", dpi=150, bbox_inches="tight")
    plt.show()
    print("Saved to activation_comparison.png")


if __name__ == "__main__":
    plot_activations()

4.3.4 Parameter Count Calculator

A utility function to compute the total number of FFN parameters for both standard and SwiGLU architectures across all layers.

Python — param_calculator.py 
def ffn_param_count(
    d_model: int,
    d_ff: int | None = None,
    n_layers: int = 12,
    variant: str = "standard",
    bias: bool = True,
    multiple_of: int = 256,
) -> dict:
    """Calculate total FFN parameter count.

    Args:
        d_model: Model dimension (e.g. 512, 768, 4096).
        d_ff: FFN hidden dimension. If None, auto-computed:
              - standard: 4 * d_model
              - swiglu: (2/3) * 4 * d_model (rounded up)
        n_layers: Number of Transformer layers.
        variant: 'standard' or 'swiglu'.
        bias: Whether bias terms are included.
        multiple_of: Round d_ff to this for hardware efficiency.

    Returns:
        Dictionary with parameter breakdown.
    """
    if variant == "standard":
        if d_ff is None:
            d_ff = 4 * d_model

        # W1: (d_model, d_ff), W2: (d_ff, d_model)
        weight_params = 2 * d_model * d_ff
        bias_params = (d_ff + d_model) if bias else 0
        per_layer = weight_params + bias_params
        n_matrices = 2

    elif variant == "swiglu":
        if d_ff is None:
            d_ff = int(2 * (4 * d_model) / 3)
            d_ff = multiple_of * ((d_ff + multiple_of - 1) // multiple_of)

        # W_gate: (d_model, d_ff), W_up: (d_model, d_ff), W_down: (d_ff, d_model)
        weight_params = 3 * d_model * d_ff
        bias_params = (2 * d_ff + d_model) if bias else 0
        per_layer = weight_params + bias_params
        n_matrices = 3

    else:
        raise ValueError(f"Unknown variant: {variant}")

    total = per_layer * n_layers

    return {
        "variant": variant,
        "d_model": d_model,
        "d_ff": d_ff,
        "n_layers": n_layers,
        "n_matrices": n_matrices,
        "bias": bias,
        "params_per_layer": per_layer,
        "total_params": total,
        "total_millions": round(total / 1e6, 2),
    }


def print_comparison(d_model: int, n_layers: int):
    """Print a comparison table for standard vs SwiGLU."""
    std = ffn_param_count(d_model, n_layers=n_layers, variant="standard", bias=True)
    swi = ffn_param_count(d_model, n_layers=n_layers, variant="swiglu", bias=False)

    print(f"{'FFN Parameter Comparison':=^60}")
    print(f"  d_model = {d_model}, n_layers = {n_layers}")
    print(f"{'':-^60}")
    print(f"  {'Metric':<25} {'Standard':>14} {'SwiGLU':>14}")
    print(f"  {'d_ff':<25} {std['d_ff']:>14,} {swi['d_ff']:>14,}")
    print(f"  {'Weight matrices':<25} {std['n_matrices']:>14} {swi['n_matrices']:>14}")
    print(f"  {'Params/layer':<25} {std['params_per_layer']:>14,} {swi['params_per_layer']:>14,}")
    print(f"  {'Total params':<25} {std['total_params']:>14,} {swi['total_params']:>14,}")
    print(f"  {'Total (millions)':<25} {std['total_millions']:>14.2f} {swi['total_millions']:>14.2f}")
    print(f"{'':=^60}")


if __name__ == "__main__":
    # Base Transformer
    print_comparison(d_model=512, n_layers=6)
    print()
    # GPT-2 scale
    print_comparison(d_model=768, n_layers=12)
    print()
    # LLaMA-7B scale
    print_comparison(d_model=4096, n_layers=32)
Implementation Notes
  • Modern LLMs typically set bias=False in FFN layers (LLaMA, Mistral, Gemma).
  • The multiple_of rounding ensures \(d_{\text{ff}}\) is friendly to GPU tensor core sizes (typically multiples of 64 or 256).
  • In practice, nn.Linear stores weights as \((d_{\text{out}}, d_{\text{in}})\), so \(\mathbf{W}_1\) is stored as shape (d_ff, d_model).
  • Some implementations fuse w_gate and w_up into a single (d_model, 2*d_ff) matrix and split the output — this is more memory-efficient on GPUs.

4.4 Going Deeper

Sparse Mixture of Experts (MoE)

Instead of one monolithic FFN, Mixture of Experts replaces the single FFN with \(N\) "expert" FFNs and a learned router that selects a small subset (typically top-2) of experts for each token. This dramatically increases the total parameter count without proportionally increasing compute, since only a few experts activate per token. MoE is used in Switch Transformer, Mixtral, and Gemini. Modern Variants

Mechanistic Interpretability

Research in mechanistic interpretability has shown that individual FFN neurons often encode interpretable concepts. Specific neurons in GPT-2 activate for concepts like "the word after a period should be capitalized" or "this token refers to a city." The key-value memory perspective (Section 4.2.4) provides a theoretical framework for understanding why individual neurons have meaning: each neuron's key (row of \(\mathbf{W}_1\)) defines what it detects, and its value (column of \(\mathbf{W}_2\)) defines what it contributes.

Knowledge Editing

Since FFN layers store factual knowledge, researchers have developed methods to edit specific facts by directly modifying FFN weight matrices. Techniques like ROME (Rank-One Model Editing) and MEMIT target specific key-value pairs in the FFN to change what the model "knows" without full retraining.
