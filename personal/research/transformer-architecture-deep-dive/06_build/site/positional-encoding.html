Positional Encoding Deep Dive

Why position matters, how sinusoidal encodings work, and why RoPE (Rotary Position Embeddings) became the standard in modern LLMs — explained with geometric intuition and from first principles.

<\!-- ==================== INTUITION ==================== -->

2.1 Intuition Layer

Intuition

2.1.1 The Bag-of-Words Problem

Here's a surprising fact: self-attention → treats its input as a set, not a sequence. Without any position information, these two sentences produce identical attention outputs:

"Dog bites man" ← same attention as → "Man bites dog"

Why? Because self-attention only computes pairwise similarities between tokens. Swapping the order of tokens just swaps the order of outputs — the attention weights between any pair of tokens remain the same. The model literally cannot tell which word came first.

2.1.2 Permutation Invariance: A Visual Proof

Let's prove this formally. If we permute the input $X$ by a permutation matrix $P$ (which rearranges rows):

$$Q' = PX W^Q = PQ, \quad K' = PK, \quad V' = PV$$

Then the attention output becomes:

$$\text{Attention}(Q', K', V') = \text{softmax}\left(\frac{PQ(PK)^T}{\sqrt{d_k}}\right)PV = \text{softmax}\left(\frac{PQK^TP^T}{\sqrt{d_k}}\right)PV$$

Since $P$ is orthogonal ($PP^T = I$ for permutation matrices), the softmax argument becomes $P \cdot \text{softmax}(\cdot) \cdot P^T \cdot PV = P \cdot \text{Attention}(Q, K, V)$. The output is just a permuted version of the original — no new information about order has been created.

This is Critical
Without positional encoding, a Transformer is a bag-of-words model with pairwise interactions. It knows which tokens are present and how they relate pairwise, but not their order. Position encoding breaks this symmetry.

2.1.3 What Properties Should a Good Position Encoding Have?

  1. Unique representation: Different positions should have different encodings
  2. Relative position: The model should be able to determine the distance between any two positions
  3. Generalization: Should work for sequences longer than those seen during training
  4. Bounded values: Shouldn't dominate the semantic content of the embedding
  5. Efficiency: Should be cheap to compute, ideally without extra parameters
<\!-- ==================== FORMALISM ==================== -->

2.2 Formalism Layer

Formalism

2.2.1 Sinusoidal Positional Encoding

The original Transformer uses sinusoidal functions at different frequencies:

$$PE(pos, 2i) = \sin\left(\frac{pos}{10000^{2i/d_{\text{model}}}}\right)$$ $$PE(pos, 2i+1) = \cos\left(\frac{pos}{10000^{2i/d_{\text{model}}}}\right)$$

where $pos$ is the position (0, 1, 2, ...) and $i$ is the dimension index (0, 1, 2, ..., $d_{\text{model}}/2 - 1$). Each pair of dimensions $(2i, 2i+1)$ forms a sinusoid with a different frequency.

Frequency Band Decomposition

Think of it like this: each dimension pair is a "clock" ticking at a different speed:

  • Dimensions 0-1: very fast oscillation (frequency = 1) — changes every position
  • Dimensions 2-3: slightly slower
  • ...
  • Last dimensions: very slow oscillation (frequency = $1/10000$) — barely changes over the sequence

Together, they create a unique "fingerprint" for each position — like binary encoding, but with smooth, continuous sinusoids instead of hard 0/1 bits.

Relative Position via Linear Transformation

A key property: for any fixed offset $k$, $PE(pos+k)$ can be written as a linear function of $PE(pos)$. Using the sin/cos addition formulas:

$$\begin{pmatrix} \sin(pos \cdot \omega + k\omega) \\ \cos(pos \cdot \omega + k\omega) \end{pmatrix} = \begin{pmatrix} \cos(k\omega) & \sin(k\omega) \\ -\sin(k\omega) & \cos(k\omega) \end{pmatrix} \begin{pmatrix} \sin(pos \cdot \omega) \\ \cos(pos \cdot \omega) \end{pmatrix}$$

The matrix on the right is a rotation matrix\! This means moving from position $pos$ to position $pos+k$ corresponds to a rotation in 2D — and crucially, this rotation depends only on the offset $k$, not on the absolute position. This is the seed idea that leads to RoPE.

2.2.2 Learned Positional Embeddings

A simpler alternative: just use a trainable embedding table $E \in \mathbb{R}^{L_{\max} \times d_{\text{model}}}$ where $L_{\max}$ is the maximum sequence length. The position embedding for position $pos$ is simply $E[pos]$ — a learned vector.

Used in BERT ($L_{\max}=512$) and GPT-2 ($L_{\max}=1024$).

2.2.3 Relative Position Encodings

Instead of encoding absolute position, encode the relative distance between tokens. Shaw et al. (2018) add a learnable bias $a_{ij}$ to the attention score based on the clipped relative distance $\text{clip}(j - i, -k, k)$.

T5 uses bucketed log-scale distances: nearby positions get fine-grained buckets, distant positions share coarser buckets. This is more parameter-efficient than learning a separate bias for every possible distance.

2.2.4 Rotary Position Embedding (RoPE) — Deep Dive

Why RoPE Matters
RoPE is used in virtually every modern LLM: LLaMA, Mistral, Gemma, Qwen, PaLM, GPT-NeoX. Understanding it is essential for understanding modern Transformers.

Step 1: The Goal

We want a position encoding function $f$ such that the dot product of position-encoded queries and keys depends only on their content and relative position:

$$\langle f(\mathbf{q}, m), f(\mathbf{k}, n) \rangle = g(\mathbf{q}, \mathbf{k}, m - n)$$

where $m, n$ are absolute positions. The inner product should be a function $g$ that depends on $\mathbf{q}$, $\mathbf{k}$, and the relative distance $m-n$ — not on the absolute positions individually.

Step 2: The 2D Insight

Consider just 2 dimensions of $\mathbf{q}$ and $\mathbf{k}$. The key insight: 2D rotation naturally encodes relative position.

A 2D rotation by angle $\theta$:

$$R(\theta) = \begin{pmatrix} \cos\theta & -\sin\theta \\ \sin\theta & \cos\theta \end{pmatrix}$$

Encode position $m$ by rotating the vector by $m \cdot \theta$:

$$f(\mathbf{x}, m) = R(m\theta) \cdot \mathbf{x}$$

Now compute the dot product of two rotated vectors:

$$\langle R(m\theta)\mathbf{q},\; R(n\theta)\mathbf{k} \rangle = \mathbf{q}^T R(m\theta)^T R(n\theta) \mathbf{k} = \mathbf{q}^T R((n-m)\theta) \mathbf{k}$$

Because $R(\alpha)^T R(\beta) = R(\beta - \alpha)$ (rotation matrices compose by adding angles). The dot product depends only on the relative position $(n-m)$\! This is exactly the property we wanted.

Figure 2.1: RoPE in 2D. Vectors q and k are rotated by their respective positions (m·θ and n·θ). Their dot product depends only on the relative angle (m−n)·θ, encoding relative position naturally.

Step 3: Extend to $d$ Dimensions

For a $d$-dimensional vector, pair up consecutive dimensions: $(0,1), (2,3), \ldots, (d-2, d-1)$. Apply a 2D rotation to each pair with a different frequency:

$$\theta_i = 10000^{-2i/d} \quad \text{for } i = 0, 1, \ldots, d/2-1$$

This is the same frequency schedule as sinusoidal PE\! The full rotation matrix is block-diagonal:

$$R_{\Theta,m} = \begin{pmatrix} R(m\theta_0) & & \\ & R(m\theta_1) & \\ & & \ddots \\ & & & R(m\theta_{d/2-1}) \end{pmatrix}$$

where each $R(m\theta_i)$ is a $2\times 2$ rotation matrix. The position-encoded query is $R_{\Theta,m} \cdot \mathbf{q}$.

Step 4: Efficient Implementation

We don't actually multiply by the rotation matrix (that would be expensive). Instead, use element-wise operations:

$$f(\mathbf{x}, m) = \mathbf{x} \odot \cos(m\boldsymbol{\theta}) + \text{rotate\_half}(\mathbf{x}) \odot \sin(m\boldsymbol{\theta})$$

where $\text{rotate\_half}$ swaps and negates pairs: $[x_0, x_1, x_2, x_3, \ldots] \to [-x_1, x_0, -x_3, x_2, \ldots]$.

This is just 2 element-wise multiplications and an addition — extremely efficient\!

2.2.5 ALiBi: Attention with Linear Biases

ALiBi (Press et al., 2022) takes a radically different approach: no position embedding at all. Instead, add a linear penalty to attention scores based on distance:

$$\text{score}_{ij} = \mathbf{q}_i \cdot \mathbf{k}_j - m \cdot |i - j|$$

where $m$ is a per-head slope. The slopes form a geometric sequence: $m_h = 2^{-8h/H}$ for head $h$ of $H$ total heads. Nearby tokens get small penalties; distant tokens get large penalties.

ALiBi's strength: excellent length extrapolation. Train on 1024 tokens, test on 2048+ with minimal quality loss. Used in BLOOM and MPT.

2.3 Engineering Layer

Engineering

2.3.1 Sinusoidal PE in PyTorch

Python — Sinusoidal Positional Encoding
import torch
import torch.nn as nn
import math

class SinusoidalPE(nn.Module):
    def __init__(self, d_model, max_len=5000):
        super().__init__()
        pe = torch.zeros(max_len, d_model)
        position = torch.arange(0, max_len).unsqueeze(1).float()
        div_term = torch.exp(
            torch.arange(0, d_model, 2).float() * -(math.log(10000.0) / d_model)
        )
        pe[:, 0::2] = torch.sin(position * div_term)  # even dimensions
        pe[:, 1::2] = torch.cos(position * div_term)  # odd dimensions
        self.register_buffer('pe', pe.unsqueeze(0))    # (1, max_len, d_model)
    
    def forward(self, x):
        """Add positional encoding to input embeddings."""
        return x + self.pe[:, :x.size(1)]


# Usage
pe = SinusoidalPE(d_model=512)
x = torch.randn(2, 100, 512)  # batch=2, seq_len=100
output = pe(x)  # position info added\!

2.3.2 RoPE in PyTorch

Python — Rotary Position Embedding (production-quality)
import torch

def precompute_freqs_cis(dim, max_seq_len, theta=10000.0):
    """Precompute the complex exponentials for RoPE.
    
    Returns cos and sin tensors of shape (max_seq_len, dim//2).
    """
    # Frequency for each dimension pair
    freqs = 1.0 / (theta ** (torch.arange(0, dim, 2).float() / dim))
    # Position indices
    t = torch.arange(max_seq_len).float()
    # Outer product: (seq_len,) × (dim//2,) → (seq_len, dim//2)
    freqs = torch.outer(t, freqs)
    # Return cos and sin
    return torch.cos(freqs), torch.sin(freqs)


def rotate_half(x):
    """Rearrange pairs: [x0, x1, x2, x3, ...] → [-x1, x0, -x3, x2, ...]"""
    x1, x2 = x[..., ::2], x[..., 1::2]
    return torch.stack((-x2, x1), dim=-1).flatten(-2)


def apply_rotary_emb(x, cos, sin):
    """Apply RoPE to queries or keys.
    
    Args:
        x: (batch, heads, seq_len, dim)
        cos, sin: (seq_len, dim//2) — precomputed frequencies
    """
    # Reshape cos/sin for broadcasting: (1, 1, seq_len, dim//2)
    cos = cos[None, None, :x.shape[2], :]
    sin = sin[None, None, :x.shape[2], :]
    
    # Repeat cos/sin to match full dimension
    cos = cos.repeat(1, 1, 1, 2)  # (1, 1, seq, dim)
    sin = sin.repeat(1, 1, 1, 2)
    
    # Apply rotation: x * cos + rotate_half(x) * sin
    return x * cos + rotate_half(x) * sin


# Example usage
d_model, n_heads, seq_len = 512, 8, 1024
d_k = d_model // n_heads  # 64

# Precompute (do this once, cache the result)
cos, sin = precompute_freqs_cis(d_k, seq_len)

# Apply to queries and keys
Q = torch.randn(2, n_heads, seq_len, d_k)
K = torch.randn(2, n_heads, seq_len, d_k)

Q_rot = apply_rotary_emb(Q, cos, sin)
K_rot = apply_rotary_emb(K, cos, sin)

# Verify: dot product depends on relative position
# Two tokens at positions 5 and 10 (relative distance = 5)
dot1 = (Q_rot[0, 0, 5] * K_rot[0, 0, 10]).sum()
# Two tokens at positions 20 and 25 (same relative distance = 5)
dot2 = (Q_rot[0, 0, 20] * K_rot[0, 0, 25]).sum()
# These should be close (same relative position)\!
print(f"Dot product (pos 5,10): {dot1:.4f}")
print(f"Dot product (pos 20,25): {dot2:.4f}")

2.3.3 Length Extrapolation

When models are tested on sequences longer than their training length, performance degrades. Different PE methods handle this differently:

MethodTrain LengthTest 2× LengthTest 4× Length
Learned PEWorksFails (out of range)Fails
Sinusoidal PEWorksDegradesDegrades badly
ALiBiWorksWorks wellWorks
RoPE (vanilla)WorksDegradesFails
RoPE + NTK scalingWorksWorksWorks well

NTK-aware scaling for RoPE: instead of the default base $\theta = 10000$, multiply by a scaling factor $s$: $\theta' = \theta \cdot s^{d/(d-2)}$. This spreads the frequencies to cover the longer sequence without retraining. More advanced methods like YaRN and LongRoPE push context to 128K+ tokens.

2.4 Going Deeper

The Length Generalization Problem

Even with scaling, there are fundamental challenges. Attention patterns learned at shorter lengths may not transfer — a model trained on 4K tokens hasn't learned to attend to information 100K tokens away. This is why many long-context models use continued pretraining on longer sequences.

Position Encoding in Vision

For Vision Transformers (ViT →), position is 2D (row, col of the patch). Most ViT models use learned 1D positions (treating the 2D grid as a flattened sequence), but 2D-aware encodings can improve performance.
